# Repository Overview Instructions

## Objective

Create a high-level overview that explains the purpose, scope, and structure of the repository.
The overview should provide readers with sufficient context to understand what the repository contains, why it exists, and how its major components relate to one another.

## Audience

Assume the reader is new to the repository and may have no prior knowledge of the project.

The overview should enable the reader to:

- Understand the repository's purpose.
- Understand the problem the repository addresses.
- Identify the major components.
- Understand the repository's scope and limitations.
- Decide whether the repository is relevant to their needs.

## Length

- Target **800–1,500 words**.
- Be comprehensive without becoming an implementation guide.
- Prioritize clarity over exhaustive detail.

## Required Sections

### Introduction

Provide a concise introduction that explains:

- What the repository is.
- The domain or problem it addresses.
- The intended audience.

### Purpose

Explain:

- Why the repository exists.
- The problem it solves.
- The goals and guiding principles.
- The value it provides.

Focus on motivation rather than implementation.

### Major Components

Describe the primary parts of the repository at a conceptual level.

For each major component, explain:

- Its purpose.
- Its responsibilities.
- How it relates to the other components.

Avoid implementation details such as file names, classes, or individual functions unless they are essential for understanding the architecture.

### Scope

Clearly define what the repository is intended to accomplish.

Describe:

- Included functionality.
- Intended use cases.
- Expected responsibilities.

### Out of Scope

Explicitly describe what the repository does **not** attempt to do.

Examples include:

- Features intentionally excluded.
- Problems left to external systems.
- Unsupported use cases.
- Design decisions that intentionally limit scope.

Clear boundaries help readers understand the repository's purpose and prevent incorrect assumptions.

### Architecture Summary

Provide a high-level description of how the major components interact.

Focus on concepts rather than implementation details.

The reader should understand the overall structure without needing to inspect the source code.

## Content Requirements

The overview should answer the following questions:

- What is this repository?
- Why does it exist?
- What problem does it solve?
- Who is it intended for?
- What are its major components?
- How do those components fit together?
- What is intentionally outside its scope?

## Writing Style

### Required

- Write in a formal and professional tone.
- Use clear, descriptive headings.
- Explain concepts before details.
- Present information logically from general to specific.
- Keep paragraphs focused on a single topic.

### Avoid

- Step-by-step instructions.
- Installation or setup guidance.
- API documentation.
- Function or class documentation.
- Implementation details.
- Internal development notes.
- Historical change logs.
- Marketing language.

## Level of Detail

Include:

- Purpose.
- Design intent.
- Conceptual architecture.
- Component responsibilities.
- Scope and limitations.

Avoid:

- Code examples.
- Configuration details.
- File structure walkthroughs.
- Build instructions.
- Deployment procedures.
- Performance tuning guidance.

## Quality Checklist

Before finalizing the overview, verify that:

- The document is between 800 and 1,500 words.
- The repository's purpose is clearly explained.
- The motivation for the repository is evident.
- The major components are described conceptually.
- The relationships between components are clear.
- The repository's scope is explicitly defined.
- The out-of-scope section establishes clear boundaries.
- The overview avoids implementation details.
- A reader unfamiliar with the project can understand its purpose after reading the document.
